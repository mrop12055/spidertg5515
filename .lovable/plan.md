

# Comprehensive Cleanup & Improvement Plan for Per-Account System

## Summary of Findings

After reviewing the codebase, I found several areas that need cleanup and improvements to fully transition to the per-account JSON + session import system and remove remnants of the old round-robin API pool approach.

## Issues Identified

### 1. **get-batch-tasks still uses API pool (not per-account priority)**
The `get-batch-tasks/index.ts` uses `getNextApiFromPool()` exclusively instead of prioritizing per-account credentials like `get-next-task` does. This means warmup and campaign batch tasks ignore JSON-provided credentials.

### 2. **Accounts.tsx deletes from telegram_api_credentials table**
When bulk deleting accounts (lines 1017-1020), the code still attempts to delete associated API credentials from `telegram_api_credentials`. This is obsolete since:
- Per-account credentials are now stored directly in `telegram_accounts` table (`api_id`, `api_hash`)
- The shared pool table should be deprecated

### 3. **Fingerprint generation code still exists in process-account-upload**
The `generateUniqueFingerprint()` function and related India-specific device lists (400+ lines) are still present. While used as a fallback, it adds unnecessary complexity since:
- JSON files should always provide fingerprints
- If missing, it should error rather than generate

### 4. **"fingerprint_generated" task type in Logs.tsx and report-task-result**
The `fingerprint_generated` task type is still referenced in:
- `src/pages/Logs.tsx` (lines 199, 212, 644)
- `supabase/functions/report-task-result/index.ts` (lines 1973-2003)

This is dead code since fingerprints come from JSON, not generated by runners.

### 5. **SetupGuide.tsx Python code still mentions "ROUND-ROBIN API SYSTEM"**
The config.py comment (lines 16-20) still says "ROUND-ROBIN API SYSTEM" which is misleading since the system now prioritizes per-account credentials.

### 6. **api-helper.ts contains legacy pool functions**
The `_shared/api-helper.ts` still has extensive pool-based functions that are now secondary (fallback only):
- `selectNextApiCredential()`
- `selectMultipleApiCredentials()`
- `recordApiUsage()`, `recordBatchApiUsage()`
- These should be marked more clearly as "LEGACY FALLBACK"

### 7. **Process-account-upload still generates fingerprints as fallback**
This adds 400+ lines of device lists and generation code that's unnecessary if we enforce JSON must have fingerprints.

## Implementation Plan

### Phase 1: Update get-batch-tasks to Use Per-Account Priority

**File: `supabase/functions/get-batch-tasks/index.ts`**

1. Import `getApiCredentialsForAccount` from api-helper
2. For warmup tasks: Use sender's `api_id`/`api_hash` first, pool fallback
3. For campaign tasks: Use account's `api_id`/`api_hash` first, pool fallback
4. Remove direct pool fetching for per-account tasks

```typescript
// Before (current - pool only):
const senderApi = getNextApiFromPool();

// senderCreds has api_id, api_hash, api_credential_id (null if per-account)
```
